Portifo · Visual System

Design System

Tokens compiled from docs/holdings-1.html — the reference mockup for Portifo's Holdings screen — and extended into forms, sheets, and pickers in that same token language. Everything below shares the live token set, not a reinterpretation: change the theme and the whole page, including the component specimens, repaints from the same custom properties the app uses.

Color

The ground is ink and paper, not gray: a near-black warmed slightly toward brown in dark mode, a warm off-white in light mode. Brass is the single accent — it marks the one primary action or active state on a screen and never competes with itself. Gain and loss are a separate semantic pair, reserved for portfolio performance only.

.row (cash / stock / closed variants) — glyph, sym, name, val, meta (shares @ avg cost = cost basis), pnl. A stock row's val is the security's live quote, not a total — the .meta line below it is what used to be the headline number, cost basis (shares @ avg cost), since the .pnl-stack's Total line already reports how far that cost basis has moved, so a separately-computed market value would only repeat the same fact in a third form. An open position's pnl is a .pnl-stack of two .pnl lines, today's gain over total gain, each led by a .pnl-label ("Today:"/"Total:") instead of a trailing tag or arrow glyph — the label reads first, before the sign. A closed row keeps a single .pnl line, still icon-tagged: it has no "today," only the total realized figure, so val already reads "Realized" instead of a price. Cash rows omit both meta and pnl (cash has no share count and no gain/loss of its own).

Buttons

Full-width, one per screen at most for the filled variants — brass for the primary path, loss for a genuine destructive confirmation (not the lighter-weight text-only treatment a sheet's action row uses), neutral for everything secondary. Google's button stays neutral by the same "one brass thing" logic — and because Google's own guidelines require an unbranded neutral button, not a tinted one.

.btn-primary (brass) · .btn-secondary (neutral) · .btn-destructive (loss) · .btn-disabled · .btn-google — the one place a fixed brand palette overrides the system, by Google's own requirement
Lists

The other two tabs behind the tab bar — Transactions and Accounts — reuse the exact same .row anatomy as Holdings: a glyph, a two-line left column, a right-aligned value column. Nothing new at the token level, only new content in the same slots. A row you can only view keeps the Holdings row's bare shape; a row that opens a picker or settings page gets the Fields chevron, so the affordance itself tells you what tapping does. Cash direction (the sign on a transaction's own value) and realized performance (a Sell's P&L) are different questions, so they get different treatment: neutral ink for one, gain/loss for the other.

Sheets

Two shapes cover it: a selection sheet, for picking one of several options, and an action sheet, for a contextual list of things to do to whatever was tapped. Both share the same chrome — handle, scrim, rounded top — but only the selection sheet gets a brass "Done": it's the one screen in this whole system where confirming is the single primary action, so brass is correct here rather than on the segmented control above.

Screens

Twelve compositions built entirely from the pieces above — nothing new invented at the token level, save for the two-dot .onboard-steps progress indicator the Onboarding screens introduce, .glyph.member, an initials-avatar variant the Settings screens add for a person, standing in for Holdings' cash/stock glyph wherever a row is about a member instead of a position, and .field-note, a single line of quiet form-help text under a field-card, for the one screen (Add Member) that needs to explain itself in prose. All are spec'd screens with no mockup of their own yet.

Set up your investment account
This is where you'll buy and sell investments. Its balance and holdings stay in sync automatically from the transactions you record — so it always shows exactly how much you've deposited, which matters for accounts like a TFSA.
Account Name
Onboarding · Step 1 of 2 — Investment Account. Runs instead of the tab bar whenever the active portfolio has zero accounts. The badge icon is the Accounts tab's own briefcase glyph, not a new one: the icon that means "Investment Account" everywhere else in the app is what teaches it here too. .onboard-steps marks progress as two 4px brass/border bars rather than numbered circles — quieter than a stepper, and it doesn't need to hold a number since there are only ever two steps. Body copy breaks .empty-body's 32ch cap to 34ch since this is explanatory prose read once, not a recurring empty-state caption. The name field reuses Update Balance's single-row .field-card + .btn-stack shape — right-aligned mono input included, for the same reason a field is "a row waiting to be saved."
Set up your cash account
Portifo also tracks cash you're holding outside your investments. To keep you from logging every small transfer, a cash account's balances are set directly rather than built from transactions — and it can hold a separate balance for each currency.

Onboarding · Step 2 of 2 — Cash Account. Same shape as Step 1 with both .onboard-step bars now on, since step 1 is complete and step 2 is current — filled reads as "done or here," outlined as "not reached yet." Badge reuses the exact cash glyph (rect + circle) from Holdings/Transactions/Accounts rows, per the same reuse rule as Step 1. Finishing this step is what takes the portfolio from zero accounts to two, which is what ends Onboarding and reveals the tab bar underneath.

Investment Account Detail — .detail-head at Holding Detail's 36px scale (an account is a peer of a holding, not the portfolio hero) + .stat-grid + two .list sections. The hero pairs with a .ccy-pill in a .hero-row, same as Portfolio and Cash — Total Value converts every holding and balance into one display currency, so the screen needs the same in-place switcher the conversion implies. The stat-grid decomposes the hero's Total Value into its two constituents, Stock Holdings and Cash Holdings (converted to that same display currency) — the count of holdings the account happens to have isn't a fact about its value, so it doesn't belong here. Holdings rows stay bare — tapping one opens Holding Detail, "the thing itself." Cash rows are .row.inert instead: no chevron and not tappable at all, since an investment account's cash balance is a derived total (built from Deposit/Withdraw transactions on that account, per Transaction Detail's own account/currency/amount fields) rather than a value the user sets directly — same shape as Holdings, but a third state beyond "opens the thing itself" and "opens a settings action": opens nothing. The Cash divider drops the .add-btn for the same reason: there's no balance to manually start here, only transactions to record. Compare the Cash Account Detail screen below, where Cash IS the thing being directly managed, so its rows keep the chevron and the add button stays.

Manage Portfolio — reached from Settings' Portfolio row, at Investment Account Detail's scale (.detail-head with no eyebrow or hero, since a portfolio has no single value of its own — that's Holdings' job). Name sits in its own single-row .field-card, the same "a field is a row waiting to be saved" idiom as Update Balance's Balance field. The Members divider carries the screen's one .add-btn, opening Add Member, per the same rule as Cash Account Detail's Currency Balances divider. Member rows reuse .row/.radio-info with the new .glyph.member in place of the cash/stock glyph and a trailing .closed-tag reporting role instead of a value. The caller's own row (labeled "(You)") drops the Fields chevron since you can't act on your own membership from here — the same .row.inert logic an Investment Account's derived Cash rows use, "opens nothing," just for a different reason. Every other row keeps the chevron, opening the Member sheet below. A member who was added by email but hasn't signed in with that Google account yet — priya@domain.com here — reuses .row.closed's dimmed opacity exactly as a closed holding does: still a real row with a real assigned role, just not yet realized, and its .sub line reads "Pending" in place of a name because there isn't one to show until they sign in. Tapping it opens the same Member sheet as any other row, so revoking a pending row and removing an active member are one interaction, not two. Delete Portfolio closes the screen as a lone .btn-destructive, the owner-only counterpart to Update Balance's Delete Balance — shown only when the caller is Owner; anyone else sees "Leave Portfolio" as .btn-secondary in its place, and neither the add-btn nor other members' chevrons render for them, since an Editor or Viewer can see the roster but not change it.

Member sheet — opened by tapping a member's row in Manage Portfolio, combining two Components patterns in one sheet rather than inventing a third: role is a .radio-row single-select exactly like Select Account (tapping a row changes it immediately, no separate save step), with a .sub line on each stating what the role can do since, unlike an account, "Owner" isn't self-explanatory. This same sheet opens for a Pending row too — the title falls back to the email since there's no name yet — and Remove from Portfolio does the same job in both cases, deleting the row outright rather than "canceling an invite" that was never sent, so Owner never has to think about the two states as different actions. Kept as one sheet instead of Add Member's full-page treatment since every choice here acts on a member who already exists, rather than creating one.
Usage
Do
  • Set anything measured — price, shares, percent, date, ticker — in IBM Plex Mono with tabular-nums.
  • Reserve Fraunces for exactly one figure per screen: the total value it's most proud of.
  • Let brass mark a single primary action or active state at a time.
  • Use the hairline dot-grid (.hero-value-row::before) only behind the hero figure — it's a ledger-paper cue, not a general background.
  • Right-align numeric field values against their label, same as a holdings row's .val — a field is a row waiting to be saved.
  • Give a destructive action in a sheet the same --loss used for negative P&L — it's already established as "this is bad" in the same product.
  • Keep third-party auth buttons (Google) in their required neutral style — it's both their brand rule and ours: brass stays reserved for our own primary action.
  • Add the Fields chevron to a list row only when tapping it opens a picker or a settings-style page (Accounts, a member row in Manage Portfolio) — leave it off a row that opens the thing itself (Holdings, Transactions), so the affordance stays honest.
  • Give a Sell transaction its own .pnl.gain/.pnl.loss line for realized P&L — the same component Holdings uses for a closed position, since a sale is the one transaction type that actually has a performance result.
  • Keep Transaction Detail strictly read-only — plain .field-card/.field rows with no chevrons — and route editing through one explicit Edit action, so opening a transaction to look at it never risks changing it.
  • Make a .row fully .inert — no chevron, no tap handler — when the value it shows is derived rather than owned, like an Investment Account's Cash rows (a running total of Deposit/Withdraw transactions), or when acting on it doesn't make sense for who's looking at it, like your own row in Manage Portfolio's member list: a third state past chevron-for-settings and bare-for-opens-the-thing-itself, for "opens nothing, because there's nothing here to open."
Don't
  • Set headings, labels, or body copy in Fraunces — it reads as a receipt total, not a title.
  • Borrow a categorical color (--c-1…--c-4) for gain/loss or vice versa — the two systems answer different questions.
  • Apply brass to more than one resting element on a screen — it stops meaning "primary" the moment it's everywhere.
  • Use a flat gray instead of --fg-3 — every neutral here carries a slight warmth from the ink/paper ground.
  • Left-align or center form input text — it breaks the tabular column a value will join once it's part of a list.
  • Put a brass "Done" on an action sheet — actions aren't a single confirmation, so nothing there should read as the one primary path.
  • Reach for an emoji or stock illustration in an empty state — the line-icon-in-a-brass-tint-circle is the system's one illustration idiom.
  • Color a transaction's cash value (.val) by gain/loss, even on a losing Sell — the proceeds are still positive cash received; the loss lives only in the separate .pnl line below it.
  • Offer a manual Update Balance path for a value the product already keeps in sync — an Investment Account's Cash rows and their divider's .add-btn stay off; a second way to set the same number is a way for it to drift from the transactions that actually back it.
